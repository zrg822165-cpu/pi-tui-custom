# OpenCode TUI 性能策略

> 这份文档基于本机 `opencode` 安装、启动包装器、配置和日志形态整理。
> 由于主程序是原生二进制，核心 TUI 源码不可直接读到，所以这里提炼的是可观察到的行为模式和可迁移的优化方向。

## 观察结论

1. `opencode` 的 CLI 入口只是一个很薄的启动器，真正的 TUI/事件循环在平台二进制里。
2. 本地配置主要负责插件、指令注入、权限和 provider wiring，TUI 逻辑不靠大量 JS 层堆叠。
3. 日志/上下文目录按会话和时间切片写入，说明它更偏向事件驱动与增量落盘，而不是整屏反复重建。
4. 性能体验的关键不在“做更多装饰”，而在“把高频变化压到最小刷新面”。

## 可迁移策略

### 1. 把高频状态做成单独 live line

- thinking / status / token 计数只能有一个活跃实例。
- 这类内容只更新当前行，不重绘整棵消息树。
- 状态文本要稳定，尽量只替换少量字段。

### 2. 工具流做聚合，不做卡片堆叠

- 连续工具调用合并成同一个 flow。
- 折叠态只保留一条标题行和一条 `⎿` 子行。
- 完成态更新摘要，不新增额外块。

### 3. 刷新路径分层

- 高频动画走 `patch` / `line replace`。
- 低频结构变化才触发局部 `requestRender`。
- 全屏重绘只能留给切换会话、展开/折叠、清屏这类操作。

### 4. 缓存渲染结果

- 组件输出按脏标记缓存。
- 子节点变化只污染局部组件，不传染整个容器。
- 纯文本状态避免每帧重新拼接复杂颜色串。

### 5. 降低动画成本

- spinner、渐变、闪烁、字级 shimmer 都要节流。
- 视觉变化优先用低频帧和少量颜色档位。
- 不要在每个 tick 里重算一整段富文本。

### 6. 把 expensive decoration 变成按需展开

- 默认态只保留信息密度最高的内容。
- 详细参数、完整工具输出、原始文本放到展开后。
- 首屏渲染越短越好。

### 7. 统一生命周期

- 一个 assistant turn 内只维护一条 thinking/status。
- 用户新输入、assistant 明确结束、会话切换时再清理。
- 避免多个 timer 并行抢渲染。

### 8. 建立性能观测开关

- 增加帧耗时日志。
- 记录 `render / diff / patch / fullRedraw` 的耗时。
- 先量化，再改动动画和布局。

## 当前项目优先级建议

1. 先压 `ThinkingStatusComponent` 的 tick 成本。
2. 再压 `ToolFlowSummaryComponent` 的折叠渲染。
3. 然后清理 `ToolExecutionComponent` 的重复布局和无效中间态。
4. 最后给 TUI 加帧耗时日志，确认优化是真降耗，不是只变得更安静。

## Claude Code WSL 观察

### 可见实现特征

- `claude` 在 WSL 里是一个很薄的启动包装，真正执行体是平台原生二进制。
- 安装包只保留 `bin/claude.exe`、`cli-wrapper.cjs`、`install.cjs` 和少量声明文件，JS 层非常薄。
- `~/.claude/settings.json` 默认是 `dark-ansi` 主题，不走重型富 UI 主题。
- 环境里显式关了非必要流量：`CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`。
- 会话记录是事件式 JSONL，而不是整屏状态快照。
- 文件变更历史按单文件、单版本拆分，像 `file-history/<uuid>/<file>@vN` 这种粒度。

### 对性能的含义

- 渲染层应该更偏文本流，而不是卡片树。
- 高成本装饰要尽量收敛到少数 live line。
- 工具、思考、文件历史都适合做增量更新，不适合整块重建。
- 持久化侧也遵循“小对象、多文件、按需读取”，可以直接作为我们这边缓存设计的参考。

### 可以直接借鉴的点

- 默认只渲染最小可用信息，不在首屏堆完整细节。
- 事件驱动刷新，结构变了才重排，高频状态只 patch 当前行。
- 跟踪数据分层存放，避免一个大对象持续膨胀。
- 关闭一切与主交互无关的附加流量和装饰。
